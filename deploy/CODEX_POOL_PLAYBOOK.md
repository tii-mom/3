# Codex 号池操盘手册（3API / Sub2API）

面向：只做 Codex 中转、中国区无法直开官方订阅、以 JSON 反代号为主力的运营。

站点：

- 控制台：`https://3api.shop`
- API：`https://api.3api.shop`

---

## 一、分组怎么建

在 **管理后台 → 分组（Groups）** 建两个组（名称可自定，建议固定英文 slug 方便运维）。

### 1. `codex-team`（主力 · Team K12 / 可反代 JSON）

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 名称 | `codex-team` | 对内运维名 |
| 平台 | OpenAI | Codex 走 OpenAI OAuth 账号 |
| 状态 | active | |
| 计费倍率 | 主推价（如 1.0～1.5，按你毛利调） | 用户扣余额的系数 |
| 模型范围 | 仅 Codex / 你实际放行的 GPT 模型 | 避免乱打烧额度 |
| 说明（对用户，可选） | 「标准 Codex，可能波动」 | 别写「官方保证」 |

**挂这个组的账号：**

- Team K12 类、卖家标明 **可反代 / sub2·cpa JSON**
- 尽量带 `refresh_token` / `rt`
- 优先级：**中高**（如 50～80）
- 单号并发：**1～2**
- 代理：**一号一代理**（海外，避开卖家禁止地区出口）

**挂这个组的用户 Key：**

- 默认套餐、活动套餐主推
- 用户默认并发：3～5（全站设置里控，防止一人抽干号池）

---

### 2. `codex-plus-trial`（试验 · Plus 成品网页号 JSON）

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 名称 | `codex-plus-trial` | |
| 平台 | OpenAI | |
| 状态 | active（库存为 0 时可先停用） | |
| 计费倍率 | 可更低或与 team 同价但限流更狠 | 损耗高，别亏本促销 |
| 说明 | 「试验线路，不稳定」 | 降低客诉预期 |

**挂这个组的账号：**

- 商家写 **反代不保、只保网页首登** 的 Plus JSON
- 优先级：**低**（如 10～30），避免抢主力流量
- 单号并发：**1**
- 标签/备注：`plus-day` / 购入日期 / 供应商

**挂这个组的用户 Key：**

- 仅内测、或单独低价 SKU
- **不要**默认给全体用户

---

### 3. （可选）`codex-pro` 兜底

| 配置项 | 建议值 |
|--------|--------|
| 上游 | 真 API Key / 稳定渠道 |
| 倍率 | 明显高于 team |
| 用途 | 大客、team 池挂了不炸盘 |

### 4. 调度相关建议（账号级）

| 项 | 建议 |
|----|------|
| 粘性会话 | 开（Codex 多轮需要） |
| 失败切换 | 开 |
| 401/403 | 自动冷却或停用，别死磕 |
| 过期 | 无 rt 的 AT 号务必设过期时间 |
| 备注字段 | `供应商 + 购入日 + 批次 + 是否有rt` |

## 二、用户侧 Key 怎么卖

| 用户类型 | 绑定分组 | 话术 |
|----------|----------|------|
| 普通充值用户 | 仅 `codex-team` | 默认线路 |
| 内测/超低价 | `codex-plus-trial` | 明确不稳定 |
| 高价稳定 | `codex-pro` | 可后续再开 |

充值：你已跑通 **付 1 入 5**。调利润优先改 **分组倍率 + 用户并发**，不要只靠堆日抛。

客户端示例：

```bash
export OPENAI_BASE_URL="https://api.3api.shop/v1"
export OPENAI_API_KEY="sk-用户在3api生成的key"
```

（以控制台「使用 Key」弹窗为准。）

## 三、10 分钟进货 → 上线检查表

> 两种货都适用。卖家常见规则：**10 分钟验活 / 禁止囤货 / 超时不售后**。

### T+0～1 分钟：下单与收货

- [ ] 下单后立刻下载，**不要隔夜**
- [ ] 只要 **JSON**（CPA 或 sub2）；邮箱密码 txt 仅备用
- [ ] 打开 JSON，记录是否有：
  - [ ] `accessToken` / `access_token`
  - [ ] `refresh_token` / `rt`（有则优先入库 team）
  - [ ] 过期时间字段

### T+1～3 分钟：格式

- [ ] 已是 **sub2** → 直接导入
- [ ] 只有 **CPA** → 用卖家转换站转 sub2（如其商品页写的转换地址）
- [ ] 转换失败 → 换供应商或按卖家说明用 CPA 兼容工具；**Sub2API 优先 sub2 字段**

### T+3～5 分钟：代理

- [ ] 准备 **空闲海外代理**（一号一 IP 最佳）
- [ ] 后台代理状态 active
- [ ] 出口不在卖家禁止地区（常见：大陆 / 港 / 澳慎用）

### T+5～8 分钟：导入 3API

路径：

`管理后台 → 账号 → 添加/导入 → OpenAI + OAuth → Codex JSON / AT 批量导入`

- [ ] 粘贴 JSON（可多行多个）
- [ ] 绑定代理
- [ ] 选择分组：`codex-team` 或 `codex-plus-trial`
- [ ] 设置优先级 / 并发（见上表）
- [ ] 备注：供应商、批次、购入时间、有无 rt
- [ ] 点击导入并创建
- [ ] 列表里状态为可用 / active

### T+8～10 分钟：验活（必须）

- [ ] 确认账号已挂到目标分组（`account_groups` / 后台「账号 → 分组」非空）
- [ ] 用 **仅绑该分组** 的测试 API Key
- [ ] 发 1 次 Codex / OpenAI responses 短请求
- [ ] 成功：标记「已验活」时间，可进调度
- [ ] 失败：立刻停用；在卖家验活窗口内走售后（若适用）
- [ ] **超时未测：按报废处理，不要上线**

### 常见故障：Codex `503` + 日志 `no available accounts`

**不是**用户 API Key 作废，而是 **Key 绑定的分组里没有可调度上游账号**。

排查：

1. Key 的 `group_id` 是哪个分组（如 `team`）
2. 该分组在 `account_groups` 是否有 `status=active` 且 `schedulable=true` 的 OpenAI OAuth 号
3. 账号是否 `error`（如 OAuth 401 / 无 refresh_token）→ 需重新导入 JSON 或重登

Nginx `413` 与号池无关，见 `docs/GO_LIVE.md` 与 `deploy/apply-nginx-body-limit.sh`。

### 上线后 24h

- [ ] 看 401 / 402 / 403 / 429 比例
- [ ] 402 空间杀：整批下线，切换备用货源
- [ ] 死号移出分组，避免污染调度

## 四、货源选型（操盘简表）

| 货类型 | 进哪组 | 仓位 |
|--------|--------|------|
| Team K12 可反代 JSON | `codex-team` | 主力，小批量滚动 |

## 五、日清进销存（建议表格列）

| 日期 | 供应商 | 类型 | 购入数 | 单价 | 有rt | 导入成功 | 验活成功 | 当日死亡 | 备注 |
|------|--------|------|--------|------|------|----------|----------|----------|------|
| | | team/plus | | | Y/N | | | | |

每日粗算：

```
当日号成本 + 代理成本
÷ 当日该分组产生的用户扣费
= 上游成本率（目标先压到你能接受的区间再放量）
```

## 六、故障口诀

| 现象 | 动作 |
|------|------|
| 导入失败 | 查 JSON 字段、是否过期、转 sub2 |
| 导入成功但请求 401 | 停用；换号；有窗口就售后 |
| 大面积 402 | 整组冷却；怀疑空间杀；换批次/供应商 |
| 用户说慢/失败 | 先看可用号数与代理，再看是否打满并发 |
| 余额到了不能用 | 查 Key 是否绑对分组、分组下是否有 active 账号 |
