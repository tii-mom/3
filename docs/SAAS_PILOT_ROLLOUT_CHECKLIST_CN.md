# 3API 白牌 SaaS 试点租户灰度验收表

## 1. 试点范围

- 首批只接入 `2-3` 个真实租户，每个租户使用独立应用实例、数据库或 Schema、Redis 命名空间、对象存储路径和日志标签。
- 试点期固定为连续 `7` 天；期间不扩大租户数量，不开启旧返利叠加，不允许租户批发余额透支。
- 每个租户指定平台负责人和租户负责人，记录域名、套餐、资源池、批发倍率、支付渠道、预算和紧急联系方式。

## 3. 启用顺序

1. 部署租户实例但不开放公网注册，完成健康检查、日志、指标和告警验证。
2. 平台给租户批发钱包小额充值，使用批发 Key 完成一次真实模型请求和精确扣费核对。
3. 租户自备支付完成最小金额充值，确认终端资金进入租户支付账户，平台不代收。
4. 开放白名单终端用户，观察 24 小时后再开放租户自己的渠道。
5. 每次只扩大一个租户或一个流量档位，变更后至少观察 4 小时。

必须同时核对：

- `期初批发余额 + 充值 - usage_debit = 期末批发余额`。
- 相同 `request_id + api_key_id` 只产生一次批发扣费。
- 租户终端订单、支付渠道实收、终端额度和 Core 批发消耗可以逐笔追溯。
- 分销佣金、SaaS 合作佣金、用户消费额度和租户批发额度四类资金不互相转入。
- outbox 无持续增长的失败事件；Redis 中断后恢复投递且消费者按幂等键去重。

## 5. 立即停止条件

任一条件触发时停止对应租户流量，不等待到次日：

- 任意账本不守恒、重复扣费、批发钱包负数或跨租户数据泄露。
- 支付成功但未到账、同一支付回调重复入账、提现重复确认。
- 域名未验证却可绑定、伪造 Host 可命中其他租户、租户可读取 Core 上游凭证。
- 单租户故障影响 Core 或其他租户，或资源配额无法阻断异常流量。
- 5 分钟错误率超过 `5%`、批发扣费 P95 超过既定 SLO 两倍，且无法在 15 分钟内恢复。

停止后先关闭租户状态或 `saas_control_plane_enabled`，保存数据库、outbox 和日志快照；禁止直接修改历史流水。

## 6. 试点通过标准

- 连续 7 天十项财务门禁全部为零，无重复支付、佣金、兑换、提现和批发扣费。
- 完成最终镜像的逻辑 `pg_dump -> psql --single-transaction` 恢复演练，并对恢复库重跑只读门禁。
- 单租户实例、数据库、Redis 或支付配置故障不影响其他租户。
- 租户毛利覆盖上游、支付、佣金、赠额、基础设施和人工支持成本。
- 平台负责人、财务负责人和运维负责人共同签字后，才允许扩大租户数量或开放自动部署。
